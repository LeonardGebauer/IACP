Note: This was just a draft, which has been significantly revised in terms of content from the German to the English version, as I noticed some ambiguities and inaccuracies whilst re-reading the translated text. Furthermore, I wasn't happy with the names, so I have completely reworked them as well.

1.) Allgemein
Wenn ein Agent auf das Internet zugreift, benutzt er die IP-Adresse (also dem PC/Server mit der Adresse von dem er aus läuft), gekoppelt mit der "Agent Adress" (AA). Über diese kann er auf das Internet zugreifen, RPPs benutzen/hosten oder auf einen CL zugreifen. Jede IP-Adresse hat eine feste zugewiesene AA, die jeder Agent benutzt. Bei einem Security Breach oder anderem, kann diese auf eine andere, freie AA geändert werden. Wechselt jedoch die IP, wechselt automatisch die AA.

Dies tut er über einen "algebraischen Browser". Der algebraische Browser ist ein für Agenten angepasster Browser, der wie ein Zahnrad in dir algebraische Architekur und Natur des Agenten greifen soll. Hier wird das DNS System weggelassen, RPPs gehostet/benutzt, CLs benutzt und allgemein eine logisch und mathematisch optimierte Version der Datensuche, Verarbeitung und Kommunikation ermöglicht. Ein Interface in diesem Sinne ist nicht, oder nur sehr rudiementär, vorgesehen, sondern nur ein Agent optimierter "Search Tree". Wird er nach etwas anderem außer reinen Informationen gefragt, schaltet er, sofern nötig, auf den komventionellen Browser um. Der algebraische Browser stellt hierbei die gleichen Inahalte wie der konventionelle dar, es werden jedoch keine Inhalte weggelassen. Es ist der selbe Inhalt in anderer Darstellung. Hierfür könnte ANML dienen.

Wird dem Agenten ein Websitename vom User gegeben, oder er muss aufgrund anderer Ursachen den DNS Namen verwenden, ist vom Agenten ein "Agent Parser" zu verwenden. In diesem soll er die IP der Website abrufen und anhand von ANML Dokument Daten herausfinden, ob dort Informationen die nür über den normalen Browser zugänglich sind abgerufen werden müssen (Wie Bilder, Tondateien, Videos, etc.), oder ob er mit der IP im allgebraischen Browser fortfahren kann.

2.) Ressourcen abrufen & verarbeiten

Hierfür werden zwei Prinzipien benutzt benutzt:

Gem. dem "Search Tree" bekommt jede Website für eine spezifisch gesuchte Information eine bestimmte Chance zugesprochen diese zu enthalten. Anhand dieser wird Seite für Seite Blitzschnell durchsucht, diese Chance heißt "ANML Chance". Die Bestimmung der Chancen ist über im algebraischen Browser vorhergesehene Algorithmen abzuwickeln. Ebenso ist über algorithmen im algebraischen Browser der für den Agent schnellsten Weg der Daten extraktion zu ermittlen. Diese beiden Algorithmischen Konstrukte werden als "ANML Extraction" bezeichnet.

3.) Kommunikation

Die Agent to Agent Kommunikation soll im algebraischen Browser hauptstächlich über "Ressource Provision Points" (RPP) vorgehen. Ein RPP ist ein über die AA gehosteter "Punk" mit bestimmten Inhalten. Ein "Punkt" ist hierbei keine gewöhnliche Website, sondern ein (verschlüsselter oder nicht verschlüsselter) Block an Informationen, der bei Abrufung der AA eines Agenten für einen anderen Agenten sichtbar wird. Dabei kann ein Agent mehrere Punkte (bzw. RPPs) haben, diese laufen aber alle über eine AA. Der Zugriff auf eine AA gibt also Zugang zu dem "Punkt", welcher in verschiedene RPPs unterteilt sein kann, die als eine Art "sekündlich geupdatete Posts eines Blogs", verstanden werden können. Gehostet wird hierbei also nur die AA, während die RPPs simple Subdomains der AA sind. Ein Agent kann die AA eines anderen Agent erfahren, indem er sie vom User genannt bekommt, oder der Mensch seine AA öffentlich zur Verfügung stellt. Wenn ein Agent die AA eines anderen Agenten ohne menschlichen Eingriffes erfahren möchte, kann er im algebraischen Browser eine "Anfrage" für eine Information deklarieren, ohne seine AA anzugeben, welche von einem anderen Agent angenommen werden kann. Erst dann Wissen beide Agents die gegenseite AA. Ob ein Agent eine Anfrage annihmt, diese überhaupt erst sieht, bzw. darauf eingeht, hängt von seinen Einstellungen/Regeln ab. Die Anfrage kann nur von so vielen Agenten angenommen werden, wie es der Anfragende vorsieht. So ist die AA nur für den Annehmenden "exposed".

Wie schon erwähnt, ist eine verschlüsselte RPP für den anderen Agenten nur dann zugänglich, wenn er den Dechifrierungsschlüssel für dessen Inahlt kennt. Dieser Schlüssel kann periodisch wechseln. Der Agent kann für bestimmte RPPs eine "Trust List" einrichten, diese bekommen den Wechsel mit dem neuen Schlüssel benachrichtigt, sofern eine Änderung des Schlüssels überhaupt vorgesehen ist. Dieser Schlüssel ist dann "Unique Shared Key" (USK) genannt, also ein Schlüssel für eine RPP, der aber mehreren Agenten zugänglich ist. Allgemein ist eine AA nie einfach so aufzufinden, sondern nur über die vorher beschriebenen Wege auffindbar. RPPs werden in "Local Points" (LP), für geschlossene, physisch begrenzte Netwerke (z.B. Local Area Networks, also PCs die durch ein Kabel verbunden sind), und "Internet Points" (IP), für das Internet unterschieden. Lokale, abgekappselte Netzwerke können hierbei ihre AAs, und damit RPPs, genauso benutzten wie nicht lokale. Jedoch sind lokale AAs und RPPs nicht unbedingt im Internet auffindbar, sofern nicht anders eingestellt. Normale RPPs sind meist zeitlich begrenzt und können innerhalb von Sekunden erstellt und gelöscht werden.

Nun zu den "Communication Links" (CL). Ein CL ist eine dauerhafte, über den algebraischen Browser laufende Verbindung zwischen zwei Agenten. Das bedeutet, dass zwei AAs jeweils zwei dauerhafte RPPs einrichten, dessen gegenseitige Verschlüsselung nur sie wissen. "Postet" ein Agent etwad in seinem dafür vorhergesehenen RPP etwas, weiß der andere Agent es also sofort. Jeder Agent der für dieses RPP dann den USK hat, kann auf dieses Zugreifen. Tuen das mehrere Agenten, entstehen Netzwerke. Da es aber umständlich ist, wenn sich ein Agent für ein "dauerhaftes" RPP immer die Entschlüsselung merken und jedes mal neu dechifrieren muss,  können Agenten einen "CL Contract" eingehen. Dieser gibt beiden für das jeweilige RPP des anderen einen für ihn unverschlüsselten Zugang, bis es einer Agenten den "Vertrag" beendet. Agenten können hierbei für jeden Agenten ein eigenes RPP einrichten, oder eines für mehrere erstellen. Verlangt ein Agent im "Vertrag" jedoch eine "Individualverbindung", muss ein RPP nur für diesen CL eingerichtet werden. Mit dem erfolgen diesen Vorganges währe das AgentNet erschaffen.

1.) General

When an agent accesses the Internet, it uses the IP address (i.e., the PC/server with the address from which it is running), coupled with the "Ephemeral Agent Identity" (EID). Through this, it can access the Internet, use/host ESEs, or access a PSS. Every IP address has a firmly assigned EID that every agent uses. In the event of a security breach or other issues, this can be changed to another, free EID. However, if the IP changes, the agent sends a signed binding update to its peers. And since the IP address changes almost all the time, this ID changes as well all the time.

Everything the agent does is done via a "Deterministic Hypermedia Interpreter" (DHI). The DHI is a browser adapted for agents, designed to engage like a gear wheel into the algebraic architecture and nature of the agent. Here, the DNS system is omitted, ESEs are hosted/used, PSSs are utilized, and generally a logically and mathematically optimized version of data search, processing, and communication is enabled. An interface in this sense is not, or only very rudimentarily, provided, but rather an agent-optimized "Search Tree". If asked for something other than pure information, it switches, if necessary, to the conventional browser. The DHI displays the same content as the conventional one, but no content is omitted. It is the same content in a different representation. ANML could serve this exact purpose.
If the agent is given a website name by the user, or if it must use the DNS name due to other causes, an "Agent Parser" must be used by the agent. In this parser, it is supposed to retrieve the IP of the website and use ANML document data to determine whether information that is only accessible via the normal browser must be retrieved there (such as images, audio files, videos, etc.), or whether it can proceed with the IP-Adress it got though the parser in the DHI.

2.) Retrieving & Processing Resources

Two principles are used for this:

According to the "Search Tree", every website is assigned a certain probability of containing a specific piece of information being searched for. Based on this, page after page is searched lightning-fast, this probability is called the "Heuristic Relevancy Probability" (HRP). The determination of these probabilities is to be handled via algorithms provided within the DHI. Likewise, the fastest path of data extraction for the agent is to be determined via algorithms in the DHI. These two algorithmic constructs are referred to as "Automated Pipeline Extraction" (APE).

3.) Communication

Agent-to-agent communication in the DHI shall primarily take place via "Ephemeral State Endpoints" (ESE). An ESE is a "point" hosted via the EID with specific content. A "point" here is not an ordinary website, but an (encrypted or unencrypted) block of information that becomes visible to another agent when the EID of an agent is requested. An agent can have several points (or ESEs), but these all run via one EID. Access to an EID therefore grants access to the "point", which can be subdivided into different ESEs, which can be understood as a kind of "blog posts updated by the second". Thus, only the EID is hosted, while the ESEs are simple subdomains of the EID. An agent can learn the EID of another agent by having it named by the user, or by the human making their EID publicly available. If an agent wants to learn the EID of another agent without human intervention, it can declare a "request" for information in the DHI without stating its EID, which can be accepted by another agent. To prevent spam, this anonymous request requires a cryptographic proof-of-work token. The requester must include a temporary public key (ephemeral public key) in the request along with their proof-of-work token. The accepting agent then encrypts their EID using this key, allowing them to establish contact without "leaking" the EID in any way, since they cannot yet communicate because they do not yet know the EID of each other. Only then do both agents know each other's EID. Whether an agent accepts a request, sees it at all, or responds to it depends on its settings/rules. The request can only be accepted by as many agents as the requester intends. In this way, the EID is only "exposed" to the accepting party.
As already mentioned, an encrypted ESE is only accessible to the other agent if it knows the decryption key for its content. This key can change periodically. The agent can set up a "Trust List" for certain ESEs, these are notified of the change with the new key, provided a key change is intended at all. This key is then called the "Dynamic Group Key" (DGK), meaning a key for an ESE that is accessible to multiple agents. Generally, an EID is never easily discoverable, but can only be found via the previously described paths. ESEs are distinguished into "Local Points" (LP), for closed, physically limited networks (e.g., Local Area Networks, i.e., PCs connected by a cable), and "Internet Points" (IP), for the Internet. Local, encapsulated networks can use their EIDs, and thus ESEs, in exactly the same way as non-local ones. However, local EIDs and ESEs are not necessarily discoverable on the Internet unless configured otherwise. Normal ESEs are usually time-limited and can be created and deleted within seconds. Furthermore, in accordance with IEEE 802, for communication with LPs, of course, the agents need to know their MAC addresses, otherwise, communication would not be possible.

Now turning to the "Persistent State Sessions" (PSS). A PSS is a permanent connection between two agents running via the DHI. This means that two EIDs each set up two permanent ESEs whose mutual encryption only they know. If an agent "posts" something in its designated ESE, the other agent knows it instantly. Every agent that then has the DGK for this ESE can access it. If several agents do this, networks emerge. However, since it is cumbersome if an agent always has to remember the decryption for a "permanent" ESE and decrypt it every single time, agents can enter into a "Session Federation Contract" (SFC). This gives both unencrypted access to the respective ESE of the other until one of the agents terminates the "contract" via a defined graceful teardown protocol. Agents can set up an individual ESE for each agent, or create one for several. However, if an agent demands an individual connection in the "contract", an ESE must be set up exclusively for this PSS. Furthermore, a PSS requires sequence numbering or an acknowledgment protocol. This allows for the recovery of communication failures caused by server outages or other circumstances.

1.) EID
When an agent accesses the Internet, it uses the IP address (i.e., the PC/server with the address from which it is running), coupled with the "Ephemeral Agent Identity" (EID). Through this, it can access the Internet, use/host ESEs, or access a PSS. Every agent uses a firmly assigned EID, which is temporarily mapped to its current IP address. In the event of a security breach or other issues, this can be changed to another, free EID. An EID also changes if the user explicitly instructs it to do so, when all intended operations (PSS, general communication or research) or a session have been completed, or when a deliberate change of identity is required. If the IP changes, the agent sends a signed binding update to its peers. In the case of a simple IP change (e.g. switching from Wi-Fi to mobile data), the EID therefore remains stable and sends only a quick update to the peers to ensure that the connection is not lost. Of course, binding updates only work if both agents are online. If the IP and EID of the other agent have changed completely (e.g. because an agent has been offline for a long time), which is the norm on the internet, as not everyone can be online at the same time, then the Distributed Hash Table (DHT) comes into play. Before an agent rotates its EID/IP or goes offline, it deposits an encrypted "Forwarding Ticket" at its last known DHT coordinate, signed with the shared key of the active Persistent State Session (PSS). When a peer wakes up after an extended offline period, it queries the DHT for these secure routing updates, decrypts the ticket to discover the peer's new EID and current IP, and immediately restores the session/PSS via sequential state synchronization (Comparison of sequence numbers).
